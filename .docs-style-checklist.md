# Documentation Style Checklist

## Core Philosophy

- [ ] **Tutorial-First, Reference-Later**: Structure documentation as a linear
      tutorial, building concepts incrementally.
- [ ] **Immediate Execution**: Code examples are for running. Encourage users to
      copy, edit, and execute.
- [ ] **Linear Progression**: Start with the simplest possible example,
      introduce one concept at a time.

## Narrative Structure

- [ ] **Conceptual Explanation Before Syntax**: Define terms before using them.
- [ ] **Explain the _why_ before the _how_**: Justify library patterns or
      conventions.
- [ ] **Explicit Links**: Use internal links to related concepts and external
      resources.

## Content Presentation

- [ ] **Code-Centric**: Code blocks are the primary vehicle for explanation.
- [ ] **Minimal Runnable Examples**: Each example should be self-contained and
      executable.
- [ ] **Show Results**: Include terminal output or visual output directly below
      code blocks.
- [ ] **Inline Terminal Examples**: Use bash blocks to show command execution
      and output.
- [ ] **Include Error Messages**: Demonstrate how the library handles invalid
      input.
- [ ] **Visual Distinction**: Bold key terms (domain concepts, library names,
      technical terms).
- [ ] **Use Tip/Warning/Info callouts sparingly**.

## Technical Specifics

- [ ] **Python Versioning**: Specify minimum Python version required (Python
      3.13+).
- [ ] **Type Hints as First-Class Citizens**: Treat type hints as core
      documentation, not just implementation details.
- [ ] **Contextual Command Prompts**: Use a generic prompt (`$ `) or indicate
      the active environment.
- [ ] **Opinionated Guidance**: Provide a specific, recommended path but
      acknowledge alternatives.

## Writing Style

- [ ] **Direct and Concise**: Clear, simple sentences. Focus on practical
      outcome.
- [ ] **User-Centric**: Explain concepts from the user's perspective.
- [ ] **Active Voice**: Prefer "The library allows you to create..." over "It is
      possible to create..."
- [ ] **Address Common Points of Confusion Explicitly**.

## Section Structure

- [ ] **Tutorial Sections**: Installation and setup, basic usage examples,
      progressive feature introduction, common patterns.
- [ ] **User Guide Sections**: Advanced usage, configuration and customization,
      integration patterns, troubleshooting.
- [ ] **Reference Sections**: API reference (separate from tutorial flow),
      changelog, contributing guidelines.

## Implementation Checklist

### For Each Document

- [ ] Start with a clear goal or "Try It Now" section.
- [ ] Provide copy‑paste commands that work immediately.
- [ ] Show expected output after each command.
- [ ] Use bold for key terms and `code` for commands and file names.
- [ ] Include internal links to related topics.
- [ ] Write in active voice, address the reader as "you".
- [ ] Eliminate hedging words ("might", "could", "possibly").
- [ ] Use concrete numbers and times ("in 2 minutes", "3 steps").

### For Code Examples

- [ ] Every code block is runnable with minimal context.
- [ ] Include imports and necessary boilerplate.
- [ ] Show the result of executing the code (output, error, etc.).
- [ ] Use type hints in Python examples.
- [ ] Specify Python version requirement if relevant.

### For Terminal Commands

- [ ] Prefix commands with `$ ` for user input.
- [ ] Show both the command and its output.
- [ ] Include error messages to illustrate failure cases.
- [ ] Use comments in bash blocks to explain each step.

## Quick Reference

- **Bold** for key terms and domain concepts.
- `Inline code` for commands, file names, and literal values.
- ```python for Python code blocks.

  ```

- ```bash for terminal commands and output.

  ```

- Links: `[link text](relative/path.md)`.
- Callouts: Use Zensical admonition syntax: `!!! note`, `!!! warning`, etc.

## Tone and Voice

- Direct, concise, practical.
- Assume the reader is technically proficient but new to the library.
- Avoid humor, fluff, and unnecessary commentary.
- Focus on what the user needs to do, not what the library does internally.
